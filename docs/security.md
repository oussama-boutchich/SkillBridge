# SkillBridge — Security Architecture

> Authentication, authorization, input validation, and security best practices.

## 1. Authentication System

### 1.1 Password Handling

| Aspect | Implementation |
|--------|---------------|
| Hashing Algorithm | bcrypt via `password_hash()` |
| Hash Cost | Default (currently 10 rounds) |
| Verification | `password_verify()` |
| Storage | Hashed string in `users.password` (VARCHAR 255) |
| Transmission | Sent over HTTPS in production; plain HTTP only in local dev |

```php
// Registration — hash password
$hashedPassword = password_hash($plainPassword, PASSWORD_BCRYPT);

// Login — verify password
if (!password_verify($inputPassword, $storedHash)) {
    Response::error('Invalid email or password.', 401);
}
```

**Rules:**
- Passwords are **never stored in plain text**
- Passwords are **never logged** or included in error messages
- Minimum password length: **8 characters**

### 1.2 Session Management

| Aspect | Implementation |
|--------|---------------|
| Mechanism | PHP native sessions (`$_SESSION`) |
| Session ID | Generated by PHP (`PHPSESSID` cookie) |
| Storage | Server-side (default file-based) |
| Lifetime | Until browser closes or explicit logout |

```php
// Session configuration (in index.php before session_start)
ini_set('session.cookie_httponly', 1);     // Prevent JS access to cookie
ini_set('session.cookie_secure', 0);       // Set to 1 in production (HTTPS)
ini_set('session.use_strict_mode', 1);     // Reject uninitialized session IDs
ini_set('session.cookie_samesite', 'Lax'); // CSRF protection
session_start();
```

**Session contents after login:**
```php
$_SESSION['user_id'] = $user['id'];
$_SESSION['email'] = $user['email'];
$_SESSION['full_name'] = $user['full_name'];
$_SESSION['role'] = $user['role'];
$_SESSION['logged_in_at'] = date('Y-m-d H:i:s');
```

**Logout process:**
```php
$_SESSION = [];
session_destroy();
setcookie(session_name(), '', time() - 3600, '/');
```

### 1.3 Ban Check

Banned users are prevented from logging in:

```php
// In AuthService::login()
$user = $this->userModel->getByEmail($email);

if ($user && $user['is_banned']) {
    Response::error('Your account has been suspended.', 403);
}
```

## 2. Authorization (RBAC)

### 2.1 Role-Based Access Control

Three roles with distinct permissions:

| Resource | Student | Company | Admin |
|----------|---------|---------|-------|
| Register | ✅ | ✅ | ❌ (seeded) |
| Own profile CRUD | ✅ | ✅ | ❌ |
| View any profile | ✅ | ✅ | ✅ |
| Manage certificates | ✅ | ❌ | ❌ |
| Browse posts | ✅ | ✅ | ✅ |
| Create/edit posts | ❌ | ✅ (own) | ❌ |
| Submit applications | ✅ | ❌ | ❌ |
| Review applications | ❌ | ✅ (own posts) | ❌ |
| Accept/reject apps | ❌ | ✅ (own posts) | ❌ |
| Ban/unban users | ❌ | ❌ | ✅ |
| View analytics | ❌ | ❌ | ✅ |
| View system logs | ❌ | ❌ | ✅ |
| Delete any post | ❌ | ❌ | ✅ |

### 2.2 Middleware Enforcement

Every protected route passes through authentication and role middleware:

```php
// Route definition with middleware
$router->post('/api/posts', 'PostController@store', ['auth', 'role:company']);
```

```php
// AuthMiddleware — verifies session exists
class AuthMiddleware {
    public static function handle(): void {
        if (!isset($_SESSION['user_id'])) {
            Response::error('Not authenticated.', 401);
        }
    }
}

// RoleMiddleware — verifies correct role
class RoleMiddleware {
    public static function handle(string $requiredRole): void {
        AuthMiddleware::handle();
        if ($_SESSION['role'] !== $requiredRole) {
            Response::error('Access denied.', 403);
        }
    }
}
```

### 2.3 Resource Ownership

Beyond role checks, controllers verify that users can only modify their own resources:

```php
// In PostController::update()
$post = $this->postModel->getById($postId);
if ($post['company_id'] !== $_SESSION['user_id']) {
    Response::error('You can only edit your own posts.', 403);
}

// In CertificateController::destroy()
$cert = $this->certModel->getById($certId);
if ($cert['user_id'] !== $_SESSION['user_id']) {
    Response::error('You can only delete your own certificates.', 403);
}
```

## 3. Input Validation & Sanitization

### 3.1 Server-Side Validation

**All input is validated on the server**, regardless of client-side validation:

```php
// Registration validation
$validator = new Validator();
$validator
    ->required('email', $input['email'] ?? '')
    ->email('email', $input['email'] ?? '')
    ->required('password', $input['password'] ?? '')
    ->minLength('password', $input['password'] ?? '', 8)
    ->maxLength('password', $input['password'] ?? '', 128)
    ->required('full_name', $input['full_name'] ?? '')
    ->minLength('full_name', $input['full_name'] ?? '', 2)
    ->maxLength('full_name', $input['full_name'] ?? '', 150)
    ->required('role', $input['role'] ?? '')
    ->inList('role', $input['role'] ?? '', ['student', 'company']);

if (!$validator->passes()) {
    Response::error($validator->getErrors(), 400);
}
```

### 3.2 Input Sanitization

```php
class Sanitizer {
    public static function string(string $value): string {
        return htmlspecialchars(trim($value), ENT_QUOTES, 'UTF-8');
    }

    public static function email(string $value): string {
        return filter_var(trim($value), FILTER_SANITIZE_EMAIL);
    }

    public static function int($value): int {
        return (int) filter_var($value, FILTER_SANITIZE_NUMBER_INT);
    }

    public static function url(string $value): string {
        return filter_var(trim($value), FILTER_SANITIZE_URL);
    }
}
```

## 4. SQL Injection Prevention

**All database queries use PDO prepared statements with parameterized queries:**

```php
// ✅ CORRECT — parameterized query
$stmt = $this->db->prepare('SELECT * FROM users WHERE email = ?');
$stmt->execute([$email]);

// ✅ CORRECT — named parameters
$stmt = $this->db->prepare('SELECT * FROM posts WHERE type = :type AND status = :status');
$stmt->execute(['type' => $type, 'status' => 'active']);

// ❌ NEVER DO THIS — string interpolation
$stmt = $this->db->query("SELECT * FROM users WHERE email = '$email'");
```

**PDO configuration enforces real prepared statements:**
```php
PDO::ATTR_EMULATE_PREPARES => false,  // Use real MySQL prepared statements
```

## 5. Cross-Site Scripting (XSS) Prevention

### 5.1 Output Encoding

All user-generated content is escaped before rendering in HTML:

```php
// Server-side encoding
echo htmlspecialchars($user['full_name'], ENT_QUOTES, 'UTF-8');
```

```javascript
// Client-side encoding (in JavaScript)
function escapeHtml(text) {
    const div = document.createElement('div');
    div.textContent = text;
    return div.innerHTML;
}

// Usage
element.innerHTML = `<h3>${escapeHtml(post.title)}</h3>`;
```

### 5.2 Content Security

- JavaScript uses `textContent` instead of `innerHTML` where possible
- User-provided URLs are validated before rendering as `href` attributes
- API responses are served with `Content-Type: application/json`

## 6. Cross-Site Request Forgery (CSRF) Prevention

### 6.1 SameSite Cookie

```php
ini_set('session.cookie_samesite', 'Lax');
```

The `Lax` SameSite attribute prevents the session cookie from being sent with cross-origin POST requests, which blocks most CSRF attacks.

### 6.2 API-Based Architecture Advantage

Since the backend is a JSON API and not a form-submission-based application, CSRF is inherently mitigated:
- API requests use `Content-Type: application/json`
- Browsers do not automatically send JSON payloads from forms
- The `credentials: 'include'` requirement ensures cookies are sent only from same-origin JS
